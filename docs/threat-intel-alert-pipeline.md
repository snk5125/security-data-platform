# Threat Intelligence Alert Pipeline — Architecture and Design Rationale

## Overview

This document explains the design of the threat intelligence ingestion, correlation, and alert forwarding pipeline. It covers the original design decisions, the latency tradeoffs that emerge as data volumes grow, and the rationale for the incremental redesign.

---

## Medallion Architecture

Security telemetry flows through three tiers before becoming an actionable alert:

```
AWS Data Sources          Databricks Medallion Layers             Downstream
─────────────────         ────────────────────────────            ──────────
Feodo Tracker ──┐
Emerging        ├──► bronze.threat_intel_raw ──► silver.threat_intel_network ──┐
Threats         │    (raw feed rows)              (normalized, deduped,         │
IPsum ──────────┘                                  TTL-managed IOCs)            │
                                                                                ├──► gold.alerts ──► SNS
VPC Flow Logs ──────────► bronze.vpc_flow ─────────────────────────────────────┘    (correlated       (forwarded
(CloudTrail,               (OCSF-formatted                                           detections)       alerts)
 GuardDuty, Config          network events)
 follow same pattern)
```

Each tier has a distinct purpose:

| Tier | Purpose | Retention |
|---|---|---|
| Bronze | Raw, append-only ingestion — preserves source fidelity | 14 days (TI), indefinite (flow) |
| Silver | Normalized, deduplicated, TTL-managed | Active indicators only (expires_at > now) |
| Gold | Correlated detections — one row per (instance, IOC) pair | Full history (append/merge) |

---

## The Two Data Planes

The pipeline joins two fundamentally different data sources. Understanding their natural update cadences is the key design constraint.

### Threat Intelligence Feeds

Feodo Tracker, Emerging Threats, and IPsum are static files published by external organizations. Each feed is regenerated by its publisher once per day. Fetching a feed more frequently than daily yields identical data and wastes compute.

- **Natural cadence:** Once per day
- **Volume per run:** ~10–20K IOC rows across all feeds
- **Silver table behavior:** MERGE on `ioc_id` — refreshes `last_seen_at` for existing indicators, inserts new ones, hard-deletes expired ones (at 2× TTL)

### VPC Flow Logs

VPC Flow Logs aggregate in 10-minute windows and are delivered to S3 continuously. New flow records are available throughout the day as EC2 instances generate network traffic.

- **Natural cadence:** Every 10 minutes (matching flow log aggregation window)
- **Volume per run:** Hundreds to thousands of flow records per ingest cycle, depending on instance count and traffic patterns
- **Bronze table behavior:** Streaming append via Auto Loader

## Original Design: Monolithic Daily Job

The initial implementation combined all four pipeline stages into a single scheduled job running once per day at 01:00 UTC:

```
threat-intel-pipeline (daily at 01:00 UTC)
  Task 1: bronze_ingest     — fetch TI feeds, write to bronze.threat_intel_raw
  Task 2: silver_network    — normalize, MERGE into silver.threat_intel_network
  Task 3: gold_alerts       — correlate VPC Flow × TI, write to gold.alerts
  Task 4: forward_alerts    — publish new alerts to SNS
```

### Why this was reasonable initially

1. **Simplicity.** One job, one schedule, one place to monitor. No inter-job dependencies to manage.
2. **Full recompute is safe.** The `gold_alerts` notebook uses a 30-day lookback window and rewrites the `ti_network` partition on every run. This guarantees that the gold table always reflects the current state of the silver TI table — including any indicators that expired since the last run. A previous alert against an IP that has since been cleaned up will not persist indefinitely.
3. **Dedup is handled.** The `gold.alerts_forwarded` tracking table prevents re-delivery regardless of how many times `gold_alerts` rewrites the gold partition.

### The latency problem

The monolithic daily design introduces up to **24 hours of alert latency**. A VPC Flow record written at 01:01 UTC — one minute after the pipeline completes — will not produce an alert until the next run at 01:00 UTC the following day.

For a security use case, 24-hour alert latency is a significant gap. A C2 beacon, lateral movement, or data exfiltration event that matches a known threat intel indicator may go unnoticed for an entire day.

### Why the full recompute gets expensive over time

The `gold_alerts` notebook joins `bronze.vpc_flow` (filtered to the last 30 days) against the full active `silver.threat_intel_network` table on every run. As the VPC Flow bronze table grows — more instances, more accounts, more days of history — the scan cost increases linearly with data volume. Running this once per day is manageable. Running it 144 times per day (once per 10-minute VPC flow ingest cycle) without redesigning the query would be prohibitively expensive.

## Redesigned Architecture: Two Jobs with Aligned Cadences

The key insight is that the two data planes have different natural cadences and should be managed independently.

```
threat-intel-pipeline (daily at 01:00 UTC)
  Task 1: bronze_ingest     — fetch TI feeds, write to bronze.threat_intel_raw
  Task 2: silver_network    — normalize, MERGE into silver.threat_intel_network
  [gold_alerts and forward_alerts REMOVED from this job]

bronze-vpc-flow-ingest (every 10 minutes)
  Task 1: ingest            — Auto Loader, write to bronze.vpc_flow
  Task 2: gold_alerts       — incremental correlation, append to gold.alerts
  Task 3: forward_alerts    — publish new alerts to SNS via CDF
```

This gives each stage a schedule aligned to its data source's natural update frequency:

| Stage | Old schedule | New schedule | Latency improvement |
|---|---|---|---|
| TI feed refresh | Daily (correct) | Daily (unchanged) | None needed |
| VPC Flow correlation | Daily | Every 10 min | Up to 24h → ~10 min |
| Alert forwarding | Daily | Every 10 min | Up to 24h → ~10 min |

## Incremental `gold_alerts`: Watermark-Based Correlation

To make correlation viable at 10-minute frequency, the `gold_alerts` notebook is redesigned from a full recompute to an incremental watermark-based job.

### How it works

On each run, the notebook reads the highest `_ingested_at` timestamp already processed from the gold alerts table:

```
last_watermark = MAX(_ingested_at) from gold.alerts WHERE alert_type = 'ti_network'
```

It then restricts the VPC Flow scan to only rows ingested since that watermark:

```
flow_df = bronze.vpc_flow WHERE _ingested_at > last_watermark
```

This bounds the scan cost to the current 10-minute ingest window rather than the full 30-day history. As the bronze table grows, the per-run cost stays roughly constant.

### Write strategy: MERGE instead of overwrite

The original full-recompute design used `replaceWhere` (partition overwrite) because it was rewriting the complete result set. The incremental design appends new alert rows and uses MERGE semantics on `alert_id` to update aggregated fields (flow_count, total_bytes, last_seen_at) when the same instance hits the same IOC again within a later 10-minute window.

```
MERGE INTO gold.alerts ON alert_id
  WHEN MATCHED → UPDATE flow_count, total_bytes, total_packets, last_seen_at
  WHEN NOT MATCHED → INSERT all fields
```

### TI expiry reconciliation

The full recompute design naturally handled expired TI indicators: on each daily run, any alert whose underlying IOC had expired would not appear in the new output, so the partition overwrite would drop it. The incremental design does not remove stale alerts automatically.

To compensate, a separate **weekly reconciliation job** (or a weekly task added to the daily TI pipeline) scans `gold.alerts` for rows whose `observable_id` no longer exists in `silver.threat_intel_network` and marks them `status = 'Expired'`. This keeps the alert history accurate without requiring a full daily recompute.

## CDF-Based `forward_alerts`: Reading Only What Changed

### The original anti-join approach

The original `forward_alerts` notebook finds unforwarded alerts by left anti-joining `gold.alerts` against `gold.alerts_forwarded` on `alert_id`. This works correctly but scans the entire `gold.alerts` table on every run. As the gold table accumulates months of alerts across multiple alert types, this scan becomes the bottleneck.

### Delta Change Data Feed

Delta Lake's Change Data Feed (CDF) exposes a changelog of inserts, updates, and deletes at the Delta transaction level. When CDF is enabled on `gold.alerts`, the forwarding notebook can read only the rows that changed since the last run:

```python
spark.read.format("delta")
    .option("readChangeFeed", "true")
    .option("startingVersion", last_processed_version + 1)
    .table("gold.alerts")
    .filter(col("_change_type").isin("insert", "update_postimage"))
```

The tracking table (`gold.alerts_forwarded`) stores the Delta version number of the last successfully processed batch. On each run, the notebook reads forward from that version. The cost is proportional to the number of alert rows written since the last run — not the total table size.

### Why this matters at scale

| Table size | Anti-join cost | CDF cost |
|---|---|---|
| 50K rows | Fast | Fast |
| 5M rows | Full table scan | Only rows since last run |
| 50M rows | Slow | Still only rows since last run |

CDF cost is bounded by write throughput (how many new alerts are produced per 10-minute window), not by total table size.

## Dedup and Delivery Guarantees

The dedup guarantee is maintained across both redesigns:

1. `gold.alerts.alert_id` is a SHA-256 of `(instance_uid | ioc_value | feed_name | alert_type)`. The same instance contacting the same IOC always produces the same `alert_id`, regardless of how many times the correlation runs.
2. `gold.alerts_forwarded` is append-only. An `alert_id` is written here only after SNS confirms receipt (`MessageId` returned). It is never deleted.
3. CDF reads in `forward_alerts` filter on `_change_type = 'insert'` or `update_postimage`. An alert that already exists in `alerts_forwarded` but receives an `update_postimage` change (e.g., `flow_count` incremented) will be re-evaluated. The forwarding notebook checks against `alerts_forwarded` before publishing — updates to already-forwarded alerts are not re-sent unless the implementation explicitly chooses to send an "updated alert" notification.

## Summary of Changes

| Component | Before | After |
|---|---|---|
| TI feed refresh cadence | Daily (in monolithic job) | Daily (dedicated job) |
| VPC Flow correlation cadence | Daily | Every 10 minutes (appended to VPC flow ingest job) |
| `gold_alerts` scan strategy | Full 30-day recompute, replaceWhere | Incremental watermark, MERGE |
| `forward_alerts` scan strategy | Anti-join full table | CDF read from last processed version |
| Alert latency | Up to 24 hours | ~10–15 minutes |
| Per-run compute cost (gold_alerts) | O(30 days of VPC flow) | O(10 minutes of VPC flow) |
| Per-run compute cost (forward_alerts) | O(total alerts in gold table) | O(new alerts since last run) |
| TI expiry handling | Automatic (full recompute drops stale alerts) | Weekly reconciliation job marks expired alerts |
