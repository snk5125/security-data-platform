# Terraform Multi-Root Restructure & Multi-Cloud Readiness

**Date:** 2026-03-18
**Status:** Approved
**Scope:** Phase 1 — Restructure existing AWS monolith into multi-root architecture. Phase 2 (future) — Add Azure workload support.

## Problem Statement

The current Terraform architecture requires a 9-phase staged apply with manual intervention (editing `terraform.tfvars` between phases, running targeted applies in sequence). This creates:

1. **Operational friction** — 9 phases with manual `tfvars` edits between Phase 5 and 5.5 (two-pass external ID pattern)
2. **Monolithic state** — all 102 resources in a single state file
3. **No multi-cloud path** — every module, variable, and notebook is AWS-specific with no abstraction layer

## Goals

- Reduce apply sequence from 9 manual phases to 3 independent roots (plus bootstrap)
- Eliminate the two-pass external ID manual edit
- Establish a workload output contract that enables any cloud provider
- Reorganize notebooks so bronze is cloud-specific, silver is the OCSF normalization boundary, and gold is cloud-agnostic

## Non-Goals

- Implementing Azure support (Phase 2, separate spec)
- Migrating to a different IaC tool
- Changing the Databricks workspace or Unity Catalog architecture
- Adding an external parameter store (e.g., SSM, Consul)

### 2. Apply Sequence

The 9-phase manual sequence is replaced by 4 independent roots:

| Step | Root | Command | Dependencies |
|------|------|---------|-------------|
| 1 | `bootstrap/` | `terraform apply` | None (one-time, local state) |
| 2 | `foundations/aws-security/` | `terraform apply` | Bootstrap state bucket exists |
| 3 | `workloads/aws-workload-*/` | `terraform apply` (parallel OK) | Foundation outputs (hub role ARN not needed here per Option C — hub owns IAM roles) |
| 4 | `hub/` | `./scripts/assemble-workloads.sh && terraform apply` | Foundation + all workloads applied |

**Note on step ordering:** With Option C (IAM roles in hub), Steps 2 and 3 are truly independent — workloads do not reference foundation outputs. The `read_only_role_arn` trust policy references the hub role, which is created in Step 4. However, workloads create the read-only role with a trust policy that initially trusts only the Databricks UC master role ARN. The hub root updates the trust policy to add the hub role after creating it. This means the IAM trust chain is fully established only after Step 4 completes.

**Alternative:** If the workload read-only role must trust the hub role at creation time (current behavior), then Step 2 must complete before Step 3, and workloads use `terraform_remote_state` to read the hub role ARN from the foundation. This is a design decision to finalize during implementation planning.

**Key improvement:** Under Option C (recommended), Steps 2 and 3 are truly independent — workloads do not reference foundation outputs, so they can start as soon as bootstrap is done. All workload roots can also be applied in parallel with each other. Under the alternative approach (workloads use `terraform_remote_state` for hub role ARN), Step 2 must complete before Step 3. In both cases, the hub root (Step 4) depends on all prior steps.

**Cross-root data flow (foundation → workloads):** Each workload root declares a `terraform_remote_state` data source pointing at the foundation root's state. This provides the `hub_role_arn` needed for the read-only role trust policy. Example:

```hcl
# workloads/aws-workload-a/data.tf
data "terraform_remote_state" "foundation" {
  backend = "s3"
  config = {
    bucket = var.state_bucket
    key    = "foundations/aws-security/terraform.tfstate"
    region = var.aws_region
  }
}

# Then reference: data.terraform_remote_state.foundation.outputs.hub_role_arn
```

#### External ID Elimination

The current two-pass pattern (apply Phase 5 → copy external IDs to tfvars → re-apply) is eliminated by splitting trust policy ownership.

**Problem:** The foundation root creates the IAM roles with trust policies that include external IDs. But the external IDs are only known after the hub root creates the Databricks storage credentials. This circular dependency forced the two-pass pattern.

**Solution:** Split the IAM role lifecycle:

1. **Foundation root owns the IAM roles** — creates `aws_iam_role` resources with a baseline trust policy that trusts only the Databricks UC master role ARN (no external ID condition, no self-assume). This is sufficient for initial creation.

2. **Hub root owns the trust policy updates** — after creating `databricks_storage_credential` resources (which output the real external IDs), the hub uses `aws_iam_role` data sources to read the existing roles, then applies `aws_iam_role_trust_policy` resources (Terraform AWS provider >= 5.x `aws_iam_role_trust_policy` or inline policy document update via a `null_resource` + `aws iam update-assume-role-policy`).

The cleanest implementation uses the foundation root's IAM roles with **placeholder trust policies** (no external ID condition) and the hub root's `aws_iam_role_policy_document` + `aws_iam_role` import-free update pattern:

```hcl
# hub/providers.tf
provider "aws" {
  alias  = "security"
  region = var.aws_region
  # Targets the security account for trust policy updates
}

# hub/main.tf — trust policy update (runs after storage credentials are created)
#
# The aws_iam_role_trust_policy resource manages ONLY the trust policy
# of an existing role without importing the role itself into this state.
# This avoids state ownership conflicts with the foundation root.
#
# NOTE: aws_iam_role_trust_policy is not a real Terraform resource type.
# Implementation options (choose during planning):
#   Option A: Foundation root accepts external_id as variable, hub passes
#             it via assemble script (adds foundation to assemble flow).
#   Option B: Use terraform_data + local-exec provisioner with
#             `aws iam update-assume-role-policy` CLI call from the hub root.
#   Option C: Move IAM roles entirely to the hub root (simplest ownership,
#             but means foundation only creates S3/KMS/SNS).
#
# Recommended: Option C — hub root owns IAM roles since it owns the trust
# policy lifecycle. Foundation creates the bucket, KMS, and SNS. The hub
# creates the IAM roles with complete trust policies in a single apply
# because it has both the UC master role ARN and the Databricks-assigned
# external IDs available at plan time.
```

**Recommended approach (Option C):** Move the hub and managed-storage IAM roles from the foundation root to the hub root. The foundation root retains S3 managed storage bucket, KMS key, and SNS resources. The hub root creates the IAM roles with correct trust policies in a single pass because it has access to `databricks_storage_credential.*.external_id` at plan time.

This means `enable_self_assume` and the external ID variables are eliminated entirely — no two-pass, no manual edits.

### 3. Workload Output Contract

Every workload root — regardless of cloud provider — exports a standardized JSON structure. This is the interface between workload roots and the hub:

```json
{
  "cloud": "aws",
  "account_id": "<WORKLOAD_A_ACCOUNT_ID>",
  "alias": "workload-a",
  "region": "us-east-1",
  "storage": {
    "type": "s3",
    "bucket_name": "security-logs-workload-a-<WORKLOAD_A_ACCOUNT_ID>",
    "bucket_arn": "arn:aws:s3:::security-logs-workload-a-<WORKLOAD_A_ACCOUNT_ID>"
  },
  "read_only_role_arn": "arn:aws:iam::<WORKLOAD_A_ACCOUNT_ID>:role/security-lakehouse-read-only",
  "encryption": {
    "type": "kms",
    "key_arn": "arn:aws:kms:us-east-1:<WORKLOAD_A_ACCOUNT_ID>:key/..."
  },
  "data_products": {
    "network_traffic": {
      "format": "json",
      "path_prefix": "vpc-flow-logs/"
    },
    "management_plane": {
      "format": "json",
      "path_prefix": "cloudtrail/"
    },
    "threat_detection": {
      "format": "json",
      "path_prefix": "guardduty/"
    },
    "resource_inventory": {
      "format": "json",
      "path_prefix": "config/"
    }
  }
}
```

**Azure equivalent (Phase 2):**

```json
{
  "cloud": "azure",
  "account_id": "<subscription_id>",
  "alias": "workload-azure-1",
  "region": "eastus",
  "storage": {
    "type": "adls",
    "storage_account": "securitylogs...",
    "container": "security-logs"
  },
  "read_only_role_arn": "<service_principal_id>",
  "data_products": {
    "network_traffic": {
      "format": "json",
      "path_prefix": "nsg-flow-logs/"
    },
    "management_plane": {
      "format": "json",
      "path_prefix": "activity-log/"
    },
    "threat_detection": {
      "format": "json",
      "path_prefix": "defender-alerts/"
    },
    "resource_inventory": {
      "format": "json",
      "path_prefix": "resource-graph/"
    }
  }
}
```

**Field notes:**
- `read_only_role_arn` — The IAM role (AWS) or service principal (Azure) that the hub credential chain-assumes to read security logs from this workload's storage. On AWS, this is the read-only role created by the data-sources module. On Azure, this is the service principal with Storage Blob Data Reader on the container.
- `data_products` keys are standardized categories. Cloud-specific data sources map into these categories — the hub and notebooks consume them by category, not by AWS service name.

### 4. Handoff Mechanism — `assemble-workloads.sh`

A shell script collects `terraform output -json` from each workload root and writes `hub/workloads.auto.tfvars.json`:

```bash
#!/usr/bin/env bash
# scripts/assemble-workloads.sh
# Collects workload outputs into hub/workloads.auto.tfvars.json

set -euo pipefail

SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
REPO_ROOT="$(dirname "$SCRIPT_DIR")"
HUB_DIR="$REPO_ROOT/hub"

# Collect workload manifests into a JSON array using jq for safe construction.
workload_json="[]"
count=0
for dir in "$REPO_ROOT"/workloads/aws-workload-* "$REPO_ROOT"/workloads/azure-workload-*; do
  [[ -d "$dir" ]] || continue
  output=$(cd "$dir" && terraform output -json workload_manifest)
  workload_json=$(echo "$workload_json" | jq --argjson w "$output" '. + [$w]')
  count=$((count + 1))
done

jq -n --argjson w "$workload_json" '{"workloads": $w}' \
  > "$HUB_DIR/workloads.auto.tfvars.json"

echo "Wrote $count workload(s) to hub/workloads.auto.tfvars.json"
```

The hub root consumes this via a `variable "workloads"` of type `list(object(...))` and iterates with `for_each` to create external locations, notebook parameters, and job configurations.

### 5. Notebook Organization

Bronze is the cloud-specific ingestion boundary. Silver is the OCSF normalization boundary — all clouds MERGE into shared tables. Gold is cloud-agnostic.

| Layer | Path | Cloud-Specific? | Purpose |
|-------|------|-----------------|---------|
| Bronze | `notebooks/bronze/aws/` | Yes | Raw ingestion per AWS data source |
| Bronze | `notebooks/bronze/azure/` (Phase 2) | Yes | Raw ingestion per Azure data source |
| Silver | `notebooks/silver/` | No | OCSF normalization — all clouds merge here |
| Gold | `notebooks/gold/` | No | Analytical products, alerts, forwarding |
| Security | `notebooks/security/threat_intel/` | No | Domain-specific: TI feeds, IOC correlation |

The `security/` folder is extensible for other domain-specific use cases (e.g., `security/compliance/`, `operations/cost/`).

**OCSF normalization at silver:**
- AWS CloudTrail + Azure Activity Log both map to OCSF `management_plane` category
- AWS VPC Flow + Azure NSG Flow both map to OCSF `network_traffic` category
- Silver tables are shared (e.g., `silver.network_traffic`) — each cloud's bronze notebook MERGEs into the same target

### 6. Unity Catalog Schema Changes

Add a `security` schema alongside the existing medallion schemas:

```
security_poc (catalog)
├── bronze          # Raw ingest (cloud-specific tables)
├── silver          # Normalized (OCSF, cloud-agnostic)
├── gold            # Analytical products
└── security        # Domain tables (threat_intel, etc.)
```

Threat intel tables (`threat_intel_raw`, `threat_intel_network`) move from bronze/silver to `security.*`. This separates reference data (feeds, IOCs) from event data (telemetry).

**Migration plan for threat intel tables:**
1. Create the `security` schema in the unity-catalog module.
2. Update all notebooks that reference `bronze.threat_intel_raw` or `silver.threat_intel_network` to use `security.threat_intel_raw` and `security.threat_intel_network` respectively. Affected notebooks: `01_bronze_ingest.py`, `02_silver_network.py`, `02_gold_alerts.py` (reads `silver.threat_intel_network` for IOC correlation).
3. Data migration: Run `CREATE TABLE security.threat_intel_raw AS SELECT * FROM bronze.threat_intel_raw` (and similarly for silver) as a one-time migration step, then drop the old tables. Alternatively, accept data loss on these tables since TI feeds are re-fetched daily — a single job run repopulates them.
4. This migration is included in Phase 1 scope since it touches the same notebook paths being reorganized.

### 7. Module Changes

**Absorb `sns-alerts` into `security-account-baseline`:** The SNS topic, publisher IAM user, and access key are security account resources. Folding them into the existing foundation module eliminates a separate module and simplifies the foundation root.

**Hub consumes workload contract:** The `cloud-integration` module gains a `workloads` variable (the standardized list) and uses `for_each` to create external locations dynamically instead of hardcoding `workload_a` and `workload_b`.

Specifics:
- **`for_each` key:** workload `alias` (e.g., `"workload-a"`, `"workload-b"`). Aliases are stable strings that persist across applies — adding a new workload does not affect existing resource addresses.
- **Static resources (not `for_each`):** The hub and managed-storage **storage credentials** remain singleton resources — there is one hub credential and one managed-storage credential regardless of workload count. Only external locations are per-workload.
- **Managed storage external location:** Remains a singleton resource (it points to the managed S3 bucket in the security account, not to a workload bucket).
- **Resource address changes:** `databricks_external_location.workload_a` → `databricks_external_location.workload["workload-a"]`. These require explicit `terraform state mv` commands during migration (see State Migration section).

```hcl
# modules/databricks/cloud-integration/main.tf (new pattern)
variable "workloads" {
  type = list(object({
    alias              = string
    cloud              = string
    storage            = object({ type = string, bucket_name = string, bucket_arn = string })
    read_only_role_arn = string
  }))
}

resource "databricks_external_location" "workload" {
  for_each = { for w in var.workloads : w.alias => w }

  name            = "security-logs-${each.key}"
  url             = "s3://${each.value.storage.bucket_name}"
  credential_name = databricks_storage_credential.hub.name
}
```

**Jobs module updated:** Notebook paths change (e.g., `bronze/01_bronze_cloudtrail.py` → `bronze/aws/01_cloudtrail.py`). The hub root is at `hub/`, so the notebook source path becomes `../notebooks/` (relative to the hub root). Job parameters reference workloads by alias from the contract.

### 8. Provider Distribution

| Root | Providers |
|------|-----------|
| `bootstrap/` | `aws` (security account) |
| `foundations/aws-security/` | `aws` (security account) |
| `workloads/aws-workload-*/` | `aws` (workload account via OrganizationAccountAccessRole assume-role) |
| `hub/` | `databricks` (workspace PAT), `aws` aliased `security` (for IAM role creation per Option C) |

Each workload root has a single AWS provider configured for its specific account. The hub root has both Databricks and AWS providers — the AWS provider targets the security account to manage IAM roles that bridge Databricks to AWS.

### 9. `apply-all.sh` Orchestration

```bash
#!/usr/bin/env bash
# scripts/apply-all.sh — Full deploy sequence for all roots
set -euo pipefail

REPO_ROOT="$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd)"

echo "=== Step 1: Bootstrap ==="
(cd "$REPO_ROOT/bootstrap" && terraform init && terraform apply -auto-approve)

echo "=== Step 2: Foundation ==="
(cd "$REPO_ROOT/foundations/aws-security" && terraform init && terraform apply -auto-approve)

echo "=== Step 3: Workloads (parallel) ==="
pids=()
for dir in "$REPO_ROOT"/workloads/aws-workload-* "$REPO_ROOT"/workloads/azure-workload-*; do
  [[ -d "$dir" ]] || continue
  (cd "$dir" && terraform init && terraform apply -auto-approve) &
  pids+=($!)
done
# Wait for all workloads; fail if any fail
for pid in "${pids[@]}"; do
  wait "$pid" || { echo "ERROR: Workload apply failed (PID $pid)"; exit 1; }
done

echo "=== Step 4: Assemble + Hub ==="
"$REPO_ROOT/scripts/assemble-workloads.sh"
(cd "$REPO_ROOT/hub" && terraform init && terraform apply -auto-approve)

echo "=== All roots applied successfully ==="
```

**Error handling:** If any step fails, the script exits immediately. Partial state is safe because each root manages its own state — re-running the script retries from the failed step (Terraform is idempotent). The `-auto-approve` flag is intentional for CI/CD; interactive use should remove it.

## State Migration Strategy

### Approach

Use `terraform state mv` to move resources from the monolithic `environments/poc/` state into per-root states. This is a one-time operation.

### Mapping

| Source (environments/poc/) | Target Root | Resources |
|---|---|---|
| `module.security_account_baseline.*` (S3, KMS only) | `foundations/aws-security/` | Managed S3 bucket, KMS key |
| `module.security_account_baseline.*` (IAM roles) | `hub/` | Hub + managed-storage IAM roles (ownership moves to hub per Option C) |
| `module.sns_alerts.*` | `foundations/aws-security/` | SNS topic, publisher IAM user |
| `module.workload_a_baseline.*`, `module.workload_a_data_sources.*` | `workloads/aws-workload-a/` | VPC, EC2, CloudTrail, etc. |
| `module.workload_b_baseline.*`, `module.workload_b_data_sources.*` | `workloads/aws-workload-b/` | VPC, EC2, CloudTrail, etc. |
| `module.cloud_integration.*`, `module.unity_catalog.*`, `module.workspace_config.*`, `module.bronze_ingestion.*` | `hub/` | Databricks resources |

### Resource Address Changes (for_each migration)

When moving from hardcoded workload resources to `for_each`-keyed resources, explicit `state mv` commands are required:

```bash
# External locations: named → for_each keyed
terraform state mv \
  'module.cloud_integration.databricks_external_location.workload_a' \
  'module.cloud_integration.databricks_external_location.workload["workload-a"]'
terraform state mv \
  'module.cloud_integration.databricks_external_location.workload_b' \
  'module.cloud_integration.databricks_external_location.workload["workload-b"]'
```

These commands must run after the state is migrated to the hub root but before the first `terraform plan` in the hub root.

### Safety Protocol

1. `terraform state pull > backup-$(date +%s).tfstate` before any migration
2. `terraform plan` in each new root after migration — must show zero changes
3. If any root shows drift, roll back with `terraform state push backup-*.tfstate`
4. A `scripts/migrate-state.sh` helper automates the sequence with dry-run mode

## Risks and Mitigations

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| State migration drift | Medium | Zero-change plan validation + backup/rollback protocol |
| Notebook path changes break running jobs | Low | Pause all jobs before migration, update paths, resume |
| `assemble-workloads.sh` produces stale data | Low | Script validates each workload root has been applied recently |
| Hub `for_each` on workloads introduces plan instability | Low | Workload aliases are stable keys; additions don't affect existing resources |
| Two-provider hub (Databricks + AWS) ordering issues | Medium | Explicit `depends_on` between credential creation and trust policy update |

---

## Success Criteria

1. Full deployment requires 4 `terraform apply` commands (bootstrap, foundation, workload, hub) with no manual tfvars editing
2. `terraform plan` in each root shows zero changes after migration
3. All existing Databricks jobs run successfully after restructure
4. Adding a new workload account requires only: copy template, fill vars, apply, assemble, hub apply
5. Workload output contract is cloud-agnostic (validated by adding an Azure template in Phase 2)
